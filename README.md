# Проблема: при разборе истории болезни нельзя “додумывать”. Ошибка в терапии/датах = критичная ошибка.

Решение в этом проекте:

* итоговый `case.json` заполняется **детерминированно** (правила + словари + строгие фильтры),
* всё, что не подтверждено текстом, остаётся `null`/пустым,
* ключевые поля (особенно лечение) сопровождаются `source` (фрагмент текста),
* результат **валидируется** JSON-схемой,
* есть **QC-отчёт** и автотесты,
* ЛЛМ используется только как **ищейка пропусков** и **не изменяет** `case.json`. Создается отдельный `llm_missing_therapy.json`.

## 1) Схема данных (что именно мы собираем)

**Главный контракт**: `schemas/container.schema.json`

Итоговый файл: `case.json` — объект по этой схеме.

Основные разделы:

* `meta` — служебные поля (id кейса, язык, профиль и т.п.)
* `patient` — демография/антропометрия/сопутствующие/аллергии/сопутствующая терапия
* `diagnoses` — диагноз(ы): нозология, подтип, стадия, ICD-10, локализация и т.п.
* `biomarkers` — биомаркеры (значение без “догадок”)
* `treatment_history` — линии/эпизоды системной терапии (режим, даты, источник)
* `radiotherapy`, `procedures`, `metastases`, `extra_findings` — дополнительные находки по тексту (если поддержаны правилами)
* `quality_gate` / `qc_report.json` — качество извлечения и предупреждения


## 2) Конвейер обработки (общее)

Вход: **только DOCX** (выписка/история болезни, в т.ч. с таблицами)

Выход: папка кейса с артефактами:

* `extracted.txt` — извлечённый текст (включая таблицы)
* `coverage.json` — “сырой слой” якорей (даты/числа/упоминания)
* `case.json` — итог по схеме
* `qc_report.json` — отчёт качества (blocker/major/warn/info)
* (опционально) `llm_missing_therapy.json` — отчёт БЯМ-ищейки (для ручной проверки)

Схема конвейера:

```
DOCX
 ├─ extract_text.py      -> extracted.txt
 ├─ normalize_med_text.py-> нормализованный текст
 ├─ coverage_layer.py    -> coverage.json + coverage_report.json
 ├─ rules/* + builder.py -> case.json (строго по схеме)
 ├─ qc_validate.py       -> qc_report.json
 └─ (опц.) LLM reviewer  -> llm_missing_therapy.json (не правит case.json)
```

## 3) Подробно по этапам

### 3.1 Извлечение текста из DOCX (`src/extract_text.py`)

* читаем **абзацы и таблицы** DOCX,
* таблицы превращаем в текстовые строки (это важно для ИГХ/патологии, где данные часто в таблице),
* сохраняем результат в `extracted.txt`.

Ограничение: формат входа — **DOCX**, другие форматы не поддерживаются.

---

### 3.2 Нормализация текста (`src/normalize_med_text.py`)

Задача: привести текст к более “одинаковому” виду, чтобы правила работали стабильно.

Что делаем (примерно):

* выравниваем пробелы/переносы строк,
* приводим тире/кавычки/служебные символы к единому виду,
* нормализуем частые сокращения и записи (например, л/у, мтс, “No/№”, варианты “c/с” и т.п.),
* убираем “шум”, который ломает регулярные выражения.

Зачем это нужно:

* иначе одно и то же клиническое значение пишут десятками способов,
* без нормализации правила расползаются и дают пропуски/ложные срабатывания.

---

### 3.3 “Сырой слой” (coverage) (`src/coverage_layer.py`)

`coverage.json` — это не “клинический смысл”, а **якоря высокой полноты**:

* даты и интервалы,
* числа/единицы,
* ключевые слова (терапия, режимы, биомаркеры и т.п.),
* кандидаты на препараты и т.д.

Зачем:

* помогает контролировать “что вообще было в тексте”,
* используется для QC и для ручного аудита,
* снижает риск, что финальные правила “не увидели” важный факт.

---

### 3.4 Правила извлечения (`src/rules/*`) + сборка в JSON (`src/rules/builder.py`)

Это ядро проекта: **строгие правила** + словари (списки терминов и регулярные выражения).

Идея:

* на каждый раздел схемы есть свой модуль правил,
* каждый модуль берёт нормализованный текст и извлекает **только то, что можно подтвердить**.

Типовые модули:

* `diagnosis_rules.py` — разбор строки диагноза (стадия, подтип, локализация и т.п.)
* `therapy.py` — линии/эпизоды терапии (регимены, даты, фильтры “планируется/рекомендовано”)
* `biomarkers.py` — биомаркеры (значение не “угадывается”, а берётся только из текста)
* `comorbidities.py` — сопутствующие (строго, без автодобавления “severity”)
* `extra_findings.py` — метастазы/процедуры/лучевая и т.п. (если поддержано)

#### Про словари и регулярные выражения

Словари и регексы — это “язык” проекта:

* они приводят множество вариантов записи к общему виду,
* позволяют извлекать значения детерминированно,
* каждое правило должно быть тестируемым.

#### Про шаблон

Шаблон итогового JSON: `examples/case_empty.json`

Как используется:

* это “скелет” будущего `case.json`,
* `builder.py` аккуратно заполняет только те поля, которые удалось подтвердить,
* если факта нет или он сомнителен — поле остаётся `null`/пустым.

## 4) Достоверность (главный принцип проекта)

### 4.1 Что запрещено

* подставлять “логичные” значения, которых нет в тексте,
* писать “negative/positive”, если это не написано явно,
* добавлять лечение из “планируется/рекомендовано/рассматривается” как проведённое.

### 4.2 Как это обеспечивается в коде

1. **Правила консервативные** (лучше пропустить, чем добавить лишнее)
2. **Фильтры планов**: слова типа “рекомендовано/планируется/возможно…” режут кандидатов на терапию
3. **Лексикон препаратов** используется, чтобы не превращать дозировку в “режим”
4. `qc_validate.py` выдаёт отчёт проблем
5. `case.json` валидируется схемой
6. БЯМ-ищейка:

   * выводит “что нашла” отдельно,
   * **не правит** `case.json`,
   * каждый пункт требует **цитату**, которая реально встречается в `extracted.txt`.

## 5) LLM reviewer (ищейка пропусков терапии)

Файл: `llm_missing_therapy.json`

Зачем:

* подсветить возможные пропуски для **ручной проверки**.

Что внутри:

* `found` — всё, что БЯМ нашла в тексте (без ограничений)
* `present` — найденное, что уже есть в `case.json`
* `missing` — найденное, чего нет в `case.json` (расчёт детерминированный по нормализованному ключу)
* `dropped` — то, что модель предложила, но цитата не нашлась в тексте (антигаллюцинация)
* `stats` — краткая статистика

Важно:

* это **не правда**, а список для проверки человеком.

## 7) Как читать `qc_report.json` (файл при тестировании)

* `blocker` — нельзя использовать кейс дальше без исправления
* `major` — вероятная проблема (например, пропущена ключевая часть)
* `warn/info` — сигнал качества/полезные замечания

Рекомендуемое правило:

* кейсы с `blocker` не отдавать в дальнейший модуль (поиск/ответы).

---

## 8) Как добавлять новые извлечения

Правило разработки:

1. сначала найти **в тексте** стабильный фрагмент-доказательство,
2. добавить правило в соответствующий модуль `src/rules/*`,
3. добавить/обновить словарь (если нужно),
4. обязательно добавить регрессионный тест в `tests/`.

---

## 9) Ограничения

* Входной формат: только DOCX.
* Система не даёт медицинских рекомендаций.
* Не все нозологии/варианты оформления покрыты: расширение делается через правила + тесты. LLM используется как временный слой покрытия.
